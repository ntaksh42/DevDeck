# 仕様書: 横断・運用効率 (カテゴリ D)

作成日: 2026-06-18 / ステータス: 提案 (未着手)

## 概要

DevDeck は「自分宛の PR レビュー・作業項目・コミット・パイプライン・コード」を
ブラウザに移動せず高速に捌くデッキ型ツールである。現状は機能ごとにビューが独立して
おり、ユーザーは「今日何をすべきか」を把握するために複数ビューを巡回する必要がある。

このカテゴリは、複数組織・複数機能を横断して「やること」を 1 画面に集約し、運用効率を
最大化するための機能群をまとめる。AGENTS.md の 4 層 IPC 契約と、dense/keyboard-first の
UI 方針に従う。

## 対象項目

| # | 項目 | 概要 |
|---|---|---|
| D-28 | 統合インボックス / 通知センター | 自分宛のメンション・割当・レビュー依頼・スレッド返信を 1 画面に集約 |
| D-29 | 未読バッジ / カウント | ナビ各項目に件数バッジを表示 |
| D-30 | 同期状態の可視化 | 最終同期時刻・同期失敗・オフラインをヘッダに明示 |
| D-31 | 横断検索の対象拡大 | コマンドパレットの横断検索にコードを追加 |
| D-32 | 複数組織クロス集計ビュー | 組織を跨いだ「今日やること」サマリ |

## D-28 統合インボックス / 通知センター

### 目的
レビュー依頼・作業項目の割当/状態変化・自分へのメンション・PR スレッドへの返信を、
1 つの時系列リストに集約し、未対応のものへ即座にジャンプできるようにする。

### 取り込み元 (既存資産の再利用)
- **PR レビュー依頼**: `myReviews` 同期キャッシュ (`CachedReviewPr`)。
- **作業項目の割当 / 状態変化**: `sync.rs` の `WorkItemNotificationEvent`
  (kind = `Assigned` / `StateChanged`) を永続化して履歴化する。
- **PR の状態変化**: `PullRequestNotificationEvent`。
- **メンション**: PR / 作業項目スレッドの自分宛メンション。新規取得が必要。

### バックエンド方針
- `sync.rs` が現在イベントとして emit している通知を、SQLite の新規テーブル
  `inbox_items` に追記する (db.rs `migrate()` でスキーマバージョンを上げる)。
  - 列: `id`, `org_id`, `kind`, `subject_id`, `title`, `detail`, `web_url`,
    `occurred_at`, `read_at` (nullable)。
- 新規 IPC: `list_inbox_items` (フィルタ: 未読のみ / 種別 / 組織)、
  `mark_inbox_item_read`、`mark_all_inbox_read`。
- 通知の発火と永続化を 1 経路にまとめ、デスクトップ通知と画面内インボックスが
  同じソースを使うようにする。

### フロントエンド方針
- 新規 view `inbox` をナビ最上部に追加 (`G i` 等のショートカット)。
- 時系列の dense リスト。行から PR レビュー / 作業項目プレビューへ直接遷移。
- 既読/未読のトグル、`Enter` で開いて既読化。
- TanStack Query キー `["inbox"]`。同期完了イベントで invalidate。

### 受け入れ基準
- レビュー依頼・割当・状態変化・メンションが 1 リストに時系列で出る。
- 行を開くと該当の PR / 作業項目に遷移し、既読になる。
- デモモードでサンプルのインボックス項目が表示される。

## D-29 未読バッジ / カウント

### 目的
ナビゲーションの各項目 (My Reviews / My Items / Inbox 等) に未対応件数を表示し、
ビューを開かずに「どこに作業があるか」を把握できるようにする。

### 方針
- カウントは既存の同期キャッシュ件数 + D-28 のインボックス未読数から算出。新規の
  重い問い合わせは避け、`list_sync_states` / `list_inbox_items` の集計を再利用する。
- `NavButton` / `NavSubItem` に `badge?: number` を追加。0 のときは非表示。
- バッジは件数の意味を `aria-label` で明示 (例: "3 未読")。

### 受け入れ基準
- My Reviews / My Items / Inbox にバッジが出る。
- 同期完了でバッジが更新される。
- 件数 0 でバッジが消える。

## D-30 同期状態の可視化

### 目的
バックグラウンド同期がいつ走り、成功したか失敗したかをユーザーが把握できるようにする。
現状ヘッダの Sync ボタンは進行中スピナーのみで、最終同期時刻や失敗が見えない。

### 方針
- `list_sync_states` は既に各スコープの最終同期メタを持つ。これを使い、ヘッダの
  Sync ボタン近傍に「最終同期: N 分前」を表示。
- 直近の同期がエラーで終わった場合は警告アイコン + ツールチップにエラーメッセージ
  (`commandErrorMessage()`)。クリックで再同期。
- ネットワーク断・認証失敗 (401) を「オフライン/要再認証」として区別表示。

### 受け入れ基準
- 最終同期時刻がヘッダに出る。
- 同期失敗時に警告とエラー内容が見える。
- 再認証が必要なケースが識別できる。

## D-31 横断検索の対象拡大 (コード)

### 目的
コマンドパレットの横断検索 (`searchAll`) は現在 Work Items / Pull Requests / Commits を
返す。コード検索もここに統合し、`Ctrl+K` 1 つで全リソースに到達できるようにする。

### 方針
- `search.rs` の `SearchAllResult` に `code` を追加。1 組織あたりの上限件数を設け、
  `totals.code` で「全件表示…」へのフォールバックを既存パターンで提供。
- コードは取得コストが高いため、パレットでは件数を絞り (例: 上限 5)、本格的な検索は
  Code Search ビューに委譲する。
- パレット接頭辞に `co:` を追加 (`wi:` / `pr:` / `c:` と同系統)。

### 受け入れ基準
- パレットでコード結果がグループ表示される。
- 「全件表示…」で Code Search ビューに query が引き継がれる。
- デモモードでコード結果のサンプルが返る。

## D-32 複数組織クロス集計ビュー

### 目的
複数組織を登録しているユーザー向けに、組織を跨いだ「今日やること」を 1 画面で俯瞰する。

### 方針
- D-28 インボックスと同期キャッシュを横断集計するダッシュボード。組織別ではなく
  「自分視点」で、レビュー待ち N 件 / 自分の作業項目 N 件 / 失敗パイプライン N 件等の
  サマリカードを並べる。
- カードからそれぞれのビュー (組織フィルタ適用済み) に遷移。
- 新規の重い IPC は追加せず、既存の同期済みデータの集計で構成する。

### 受け入れ基準
- 複数組織のレビュー待ち / 作業項目 / 失敗ビルドが 1 画面に集約される。
- カードから各ビューに遷移できる。
- 組織が 1 つだけのときも破綻しない (単一組織サマリとして機能)。

## 共通の非機能要件
- すべてのインタラクティブ要素にキーボード経路を用意する (AGENTS.md hard requirement)。
- デモモード (`isTauriRuntime()` false) で破綻しないこと。新規コマンドはデモ実装と
  Zod スキーマを必須成果物とする。
- 検証: `pnpm tsc --noEmit`, `pnpm test -- --run`, `cargo test --workspace`。
