# 仕様書: 信頼性・基盤 (カテゴリ F)

作成日: 2026-06-18 / ステータス: 提案 (未着手)

## 概要

このカテゴリは、機能追加ではなく、DevDeck を日常運用に耐えるツールにするための
信頼性・配布・診断まわりの改善をまとめる。同期の取得範囲・効率、エラー回復、
配布の自動化、診断のしやすさが対象。

## 対象項目

| # | 項目 | 概要 |
|---|---|---|
| F-37 | active 以外の PR の限定取得 | completed/abandoned の履歴追跡 |
| F-38 | 同期のインクリメンタル化 | changedDate ベースの差分取得でレート最適化 |
| F-39 | エラーのリトライ UI | コマンド失敗をトースト + 再試行で回復可能に |
| F-40 | ログ / 診断のエクスポート | 不具合報告用の診断バンドル出力 |
| F-41 | 自動アップデート | Tauri updater による更新 |
| F-42 | ローカル使用統計 | テレメトリ無しの軽い達成サマリ |

## F-37 active 以外の PR の限定取得

### 背景・制約
AGENTS.md 明記の現状制約: PR ビューはローカル同期された **active** PR データから動作し、
「all statuses」のような未対応を示唆する UI は出さない方針。本項目はサービスとキャッシュ
層を同時に更新し、completed/abandoned の **限定的** な追跡を可能にする。

### 方針
- `prs.rs` / `sync_prs_for_org` に、自分が author / reviewer の completed/abandoned PR を
  期間/件数で限定取得するパスを追加 (全件同期はしない)。
- キャッシュテーブルに status と completed/closed 日時を持たせ、ビューでは明示的に
  「最近完了した PR」など限定的な表現にする (「all statuses」とは書かない)。
- My Reviews / PR Search のフィルタに「最近完了」を限定追加。

### 受け入れ基準
- 自分が関与した最近の completed/abandoned PR が限定的に閲覧できる。
- 未対応範囲を示唆する文言を出さない。
- active 同期の挙動・件数が劣化しない。

## F-38 同期のインクリメンタル化

### 目的
同期のたびに全件を取り直す箇所を、`changedDate` 等のウォーターマークによる差分取得に
変え、API レートと同期時間を削減する。

### 方針
- スコープ別 (myReviews / myWorkItems / commits) に最終同期ウォーターマークを
  `sync_states` に保持 (既存の最終同期メタを拡張)。
- 作業項目は `System.ChangedDate > @watermark` を WIQL に付与。PR/コミットも各 API の
  時刻フィルタを使う。差分が無いときは早期リターン。
- 共通の `AdoClient::get_json`/`post_json` 経由を維持し、リトライ・429 Retry-After 挙動を
  変えない。
- 削除・status 変化の取りこぼしに注意 (定期的なフル同期をフォールバックとして残す)。

### 受け入れ基準
- 変更が無い同期で REST 呼び出しが大幅に減る。
- 変更があった項目のみ更新され、既存項目が消えない。
- `cargo test --workspace` (wiremock) で差分取得が検証される。

## F-39 エラーのリトライ UI

### 目的
コマンド失敗 (ネットワーク断・5xx・429 超過) を握りつぶさず、ユーザーがその場で再試行
できるようにする。

### 方針
- ミューテーション/クエリ失敗を捕捉する共通トースト機構を追加 (`commandErrorMessage()`
  でメッセージ抽出)。トーストに「再試行」ボタンを置き、該当の mutate/refetch を再実行。
- 既存の `ErrorState` (全画面エラー) は重い失敗に残し、操作系の失敗はトーストで非破壊に
  通知する。
- 401 は再認証導線、429 は自動リトライ済みである旨を区別表示。

### 受け入れ基準
- コメント投稿・投票・状態変更などの失敗がトーストで通知される。
- 再試行ボタンで同じ操作を再実行できる。
- 401 / 429 が区別して提示される。

## F-40 ログ / 診断のエクスポート

### 目的
不具合報告のために、秘密情報を含まない診断バンドルをワンクリックで出力する。

### 方針
- 設定画面に「診断をエクスポート」を追加。アプリバージョン、OS、スキーマバージョン、
  同期状態サマリ、直近のエラー (メッセージのみ) を 1 ファイルにまとめて保存。
- **PAT / Azure CLI トークンは一切含めない** (AGENTS.md secret rules)。組織名・URL も
  秘匿が必要なら伏字オプションを用意。
- `tracing` のログがファイル出力されている場合はその場所を案内 (本文は同梱しない)。

### 受け入れ基準
- 診断ファイルが生成され保存できる。
- ファイルに秘密情報が含まれない (テストで検証)。
- バージョン・同期状態・直近エラーが含まれる。

## F-41 自動アップデート

### 目的
リリース (GitHub Actions `release.yml`、`vX.Y.Z` タグ) で配布される新版を、ユーザーが
手動再インストールせず更新できるようにする。

### 方針
- Tauri updater プラグインを導入。更新元は GitHub Releases。署名鍵をリリースフローに
  組み込む。
- 起動時 / 手動チェックで更新有無を確認し、ユーザー確認の上でダウンロード・再起動。
- 設定に「自動チェック」トグルと現在バージョン表示。

### 受け入れ基準
- 新しいリリースが検出され、確認の上で適用できる。
- 自動チェックの ON/OFF が設定できる。
- 署名検証が有効である。

## F-42 ローカル使用統計

### 目的
外部送信なしで、ユーザー自身の「捌いた件数」など軽い達成サマリをローカルに表示する。

### 方針
- ローカルのみ。レビュー投票数・解決スレッド数・状態変更数などをローカルにカウント
  (SQLite or localStorage)。**外部テレメトリ送信は行わない**。
- 設定 or インボックスに控えめなサマリを表示。リセット可能。

### 受け入れ基準
- 自分の操作件数がローカルに集計・表示される。
- 一切外部送信しない。
- リセットできる。

---

## 共通の非機能要件
- 秘密情報をログ・診断・統計・テストに含めない (AGENTS.md secret rules)。
- `azdo-client` クレートを Tauri 非依存に保つ。
- REST 挙動変更は wiremock テストを優先する。
- 検証: `cargo test --workspace`, `cargo clippy --workspace --all-targets -- -D warnings`,
  リリース確認は `pnpm tauri build`。
